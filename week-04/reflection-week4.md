## Главный инсайт недели

**Детерминированные правила → в код, не в промпт.**

Это самый важный вывод. Агент три прогона подряд нарушал band cap — даже когда сам писал "hard ceiling rule" в ответе (u006, Run 2). Промпт не побеждает обученное поведение модели на точных числовых ограничениях.

Решение: `apply_band_cap()` — post-processing в коде после каждого API вызова. 100% надёжно, 0 зависимости от промпта.

```
Промпт контролирует: тон, структуру, стиль, педагогику
Код контролирует:    числовые ограничения, форматирование, deterministic rules
```

Второй инсайт: **eval flywheel работает**. Каждый прогон давал конкретный failure mode → конкретный фикс → следующий прогон. Без harness это было бы ручной проверкой 30 кейсов после каждого изменения промпта.

## Что узнал про eval engineering

### 1. Два типа failures — два разных решения

| Тип failure | Причина | Решение |
|---|---|---|
| band cap нарушается | Промпт vs. обученное поведение | Код: apply_band_cap() |
| b6001 band range | Датасет слишком строгий | Датасет: расширить диапазон |
| e006 не детектируется | Rubric слишком узкий | Rubric: добавить паттерны |
| b7006 занижен | 189 слов = честная 6.0 | Датасет: снизить min |

### 2. Dataset — живой документ
За 4 прогона датасет менялся дважды (b6001, b7006). Это нормально — golden dataset уточняется через реальное поведение агента. Статичный датасет написанный "от головы" будет либо слишком строгим, либо слишком мягким.

### 3. Rubric важнее датасета
`check_refusal_triggered()` с 15 паттернами детектировал e006 надёжно после расширения. Узкий rubric = false negatives = ложное ощущение что агент работает.

### 4. 95%+ с первого прогона — хороший результат
80/83 на Run 1 означает что Context Pack v4 уже хорошо откалиброван. Eval harness не нашёл системных проблем — только edge cases в граничной зоне (87-140 слов).

## Архитектура финального harness

```
dataset.json (30 кейсов, 5 категорий)
    ↓
runner.py::call_agent()
    → API call → apply_band_cap() → response
    ↓
rubric.py::apply_rubric()
    → 14 code-based checks
    → score_rubric() → passed/total/failures
    ↓
pytest assert
    → PASS / FAIL с детальным сообщением
    ↓
results.json
```

**Ключевое архитектурное решение:** `apply_band_cap()` между API call и rubric — не в промпте, не в rubric, а в pipeline.

## Неделя 5 — что планирую

**Тема:** LLM-as-a-Judge

**Главный вопрос:** как автоматически проверить то, что код проверить не может — качество цитат, полезность Exercise, правильность тона?

**Что строю:**
```
judges/
  judge_citations.py    ← Цитирует ли ответ конкретный текст?
  judge_exercise.py     ← Конкретное задание vs. общие советы?
  judge_tone.py         ← Честный коуч vs. лесть или грубость?
  calibration.py        ← Judge agreement с ручной оценкой ≥ 80%?
```

**Связь с Неделей 4:** rubric.py уже имеет `check_cites_text()` — простой regex. LLM-judge заменит его на семантическую проверку с объяснением почему цитата засчитана или нет.
