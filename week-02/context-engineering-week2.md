## Главный сдвиг в мышлении

| Prompt Engineering | Context Engineering |
|---|---|
| Как написать правильные слова | Что вообще передавать модели |
| Один промпт → один результат | Итеративное управление контекстом на каждом шаге |
| Фокус на system prompt | Фокус на всём: system prompt + tools + history + data + memory |
| Подходит для one-shot задач | Подходит для агентов с длинным горизонтом |

> **Определение от Anthropic:** Context Engineering — это стратегии по отбору и поддержанию оптимального набора токенов во время LLM inference, включая всё что попадает в контекст помимо промптов.

## 3 Принципа

### Принцип 1 — Контекст это конечный ресурс (Attention Budget)
Модель, как человек, **теряет фокус** при увеличении контекста. Это явление называется **context rot** — чем длиннее контекст, тем хуже модель извлекает информацию из него. Причина архитектурная: transformer создаёт n² пар связей между n токенами — при росте контекста внимание "размазывается".

**Вывод:** каждый токен стоит денег внимания. Добавляй только то, что реально нужно.

### Принцип 2 — Минимальный набор максимально сигнального контента
Цель context engineering: найти **наименьший возможный набор высокосигнальных токенов**, который максимизирует вероятность нужного поведения. Это не значит "короткий" — это значит "без шума".

**Вывод:** начинай с минимального промпта, тестируй, добавляй только то, что исправляет конкретный failure mode.

### Принцип 3 — Just-in-Time контекст вместо предзагрузки
Вместо того чтобы загружать все данные заранее — агент держит **лёгкие идентификаторы** (file paths, ссылки, stored queries) и динамически подгружает данные в момент, когда они нужны. Claude Code так и работает: пишет запросы, использует `head`/`tail`, не загружая полные файлы.

**Вывод:** для агентов проектируй инструменты для навигации и получения данных, а не предзагрузку всего массива.

## 2 Паттерна

### Паттерн 1 — Правильная "высота" системного промпта (Right Altitude)

Два failure mode которые надо избегать:

```
TOO LOW (хрупкость)                TOO HIGH (расплывчатость)
──────────────────                 ──────────────────────────
if user says X → do Y              "Be helpful and professional"
if score < 5 → respond Z           "Use your judgment"
Hardcoded logic                    Ложное предположение shared context

              ✅ ПРАВИЛЬНО: Goldilocks Zone
              Достаточно конкретно чтобы направлять
              Достаточно гибко чтобы модель использовала эвристику
```

**Пример правильной высоты для GainScore:**
- ❌ Слишком низко: "If word count < 250 then output: ⚠️ Word count: N words — below minimum"
- ❌ Слишком высоко: "Give feedback on essays"
- ✅ Правильно: "Flag word count as priority if essay is under 250 words — this directly caps TA score"

### Паттерн 2 — Compaction для длинных задач

Когда агент накапливает длинную историю → **сжатие контекста**:
1. Передать историю модели для суммаризации
2. Сохранить: архитектурные решения, незакрытые баги, важные детали
3. Удалить: повторяющиеся tool outputs, промежуточные сообщения
4. Начать новый контекст с компактным summary + N последних файлов

**В Claude Code:** компакция сохраняет summary + 5 последних файлов. Самый безопасный вариант — очистка результатов tool calls из истории.

## 1 Антипаттерн

### ❌ Раздутый набор инструментов (Bloated Tool Set)

> "Если человек-инженер не может однозначно сказать, какой инструмент использовать в данной ситуации — AI-агент не сможет этого сделать лучше."

Признаки проблемы:
- Несколько инструментов с перекрывающейся функциональностью
- Неясные границы — когда использовать tool A, а когда tool B
- Инструменты возвращают избыточные данные (не token-efficient)

**Фикс:** минимально жизнеспособный набор инструментов. Каждый инструмент — самодостаточный, устойчивый к ошибкам, с однозначным назначением.

## Анатомия контекста агента (порядок важен)

```
┌─────────────────────────────────────────────┐
│  1. System Prompt                           │  ← Роль, инструменты, ограничения,
│     (identity, capabilities, rules,         │     паттерны поведения
│      output format)                         │
├─────────────────────────────────────────────┤
│  2. Relevant Memory                         │  ← Релевантное из прошлых сессий
│     (structured notes, past decisions)      │     (не вся история!)
├─────────────────────────────────────────────┤
│  3. Tool Definitions                        │  ← Минимальный набор, чёткие границы
│     (schemas, descriptions)                 │
├─────────────────────────────────────────────┤
│  4. Few-shot Examples                       │  ← Разнообразные канонические примеры,
│     (diverse, canonical)                    │     не список edge cases
├─────────────────────────────────────────────┤
│  5. Conversation History                    │  ← Компактная, очищенная от шума
│     (compacted, cleaned)                    │
├─────────────────────────────────────────────┤
│  6. Just-in-Time Data                       │  ← Подгружается через tools в момент
│     (retrieved at runtime)                  │     когда нужно, не заранее
└─────────────────────────────────────────────┘
```

## Техники для длинного горизонта

| Техника | Суть | Когда применять |
|---|---|---|
| **Compaction** | Сжать историю в summary, начать новый контекст | Длинные back-and-forth диалоги |
| **Structured Note-taking** | Агент пишет NOTES.md, читает при следующем запуске | Итеративная разработка с чёткими milestone |
| **Sub-agent Architecture** | Главный агент координирует, подагенты работают в чистом контексте и возвращают summary (1-2k токенов) | Параллельные исследовательские задачи |

## Context Pack — шаблон (итог Недели 2)

```markdown
## Goal
[Что должна сделать модель — одно предложение]

## User Profile
[Кто пользователь, уровень, цель, ограничения]

## Task
[Конкретный запрос с деталями]

## Constraints (Hard Rules)
- NEVER [действие 1]
- NEVER [действие 2]
- ALWAYS [обязательное поведение]

## Data / Context
[Документы, история, входные данные — только релевантное]

## Examples (Few-shot)
<example>
Input: ...
Output: ...
</example>
